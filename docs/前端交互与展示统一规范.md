# 前端交互与展示统一规范

## 1. 文档定位

本文档是 HGT 项目已经确认的跨页面交互与展示规范的唯一来源。

后续任务只要涉及以下任一事项，开始设计、修改或审查代码前必须完整阅读本文档：

- 聊天输入框、表情包、消息列表或历史消息加载
- 圈子、私信、玩汤房间
- `@用户`、未读消息、顶部消息横幅或底部导航提示
- 用户头像、昵称、在线状态
- 昵称旁的徽章图标、徽章名称
- 聊天图片、头像或表情资源的压缩与缓存

新增跨页面规则或改变本文档中的既有规则时，必须同步更新本文档，不再创建内容重叠的独立规范。

## 2. 共用实现原则

1. 同一种交互在不同页面必须复用共享组件或共享状态逻辑，禁止复制后各自演进。
2. 已有基准组件优先于页面内手写样式：
   - 表情包键盘：`apps/web/src/components/StickerKeyboard.tsx`
   - 徽章图标与名称：`apps/web/src/components/BadgeVisuals.tsx` 中的 `EquippedBadgeIcon`
   - 顶部实时消息横幅：`apps/web/src/components/IncomingMessageBanner.tsx`
   - 全局未读统计：`apps/web/src/shared/useMessageUnread.ts`
3. 页面可以根据空间调整头像或徽章图标尺寸，但不能自行改变名称颜色、字号、字重、间距和等级规则。
4. 新功能必须同时考虑手机安全区、软键盘、窄屏截断、触控误触、加载状态、失败状态和无障碍名称。
5. 相关修改至少执行：
   - `npm run check`
   - `npm run build:all`
   - 涉及后端逻辑时执行 `npm test`

## 3. 表情包键盘

### 3.1 适用范围

适用于所有允许选择并发送表情包的聊天页面：

- 玩汤房间聊天
- 私信聊天
- 圈子聊天
- 后续新增的聊天场景

消息气泡、会话摘要、顶部横幅和通知中的表情只属于只读展示，不属于表情包键盘。

### 3.2 打开与关闭

1. 点击表情包按钮时关闭系统软键盘，表情包键盘占据输入区下方空间。
2. 不使用弹窗、抽屉或 `fixed` 浮层覆盖消息列表。
3. 再次点击表情包按钮或收起按钮时关闭面板。
4. 面板打开时点击文字输入框，应关闭表情包面板并唤起系统软键盘。
5. 离开当前聊天、切换聊天对象或切换房间后，面板恢复关闭状态。
6. 表情包键盘与 `@成员` 候选等输入辅助面板互斥。
7. 切换面板不得清空已输入文字、光标位置或已经选择的 `@成员`。

### 3.3 布局

1. 所有屏幕尺寸固定为每页 4 列 × 2 行，共 8 个表情。
2. 桌面端也不能改成 6 列或更多列。
3. 单元格只展示表情图片，名称通过 `aria-label` 提供，不在图片下方常驻显示。
4. 图片使用 `object-contain`，不得裁切、拉伸。
5. 面板包含标题“表情包”、收起按钮、4 × 2 表情区和多页时的页码圆点。
6. 加载中、空状态、最后一页数量不足和切页时，表情区域高度保持稳定。
7. 面板展开后消息区同步缩小，不得被覆盖；原本位于底部时继续跟随最新消息。
8. 底部必须适配 `safe-area-inset-bottom`。

### 3.4 分页与手势

1. 按接口返回顺序展平表情系列，每 8 个切分一页。
2. 左滑进入下一页，右滑返回上一页；首尾页不循环。
3. 滑动跟手，边界具有阻尼。
4. 横向位移达到 40px 时切页，未达到时回弹。
5. 位移超过 8px 后按滑动处理，结束时不得误触发送。
6. 回弹和切页动画约 300ms。
7. 一页时不显示圆点，多页时提供“第 X 页，共 Y 页”的状态说明。
8. 横滑逻辑不得无条件阻止页面纵向滚动。

### 3.5 发送、异常和性能

1. 点击表情后立即发送，不需要二次确认；发送期间禁止重复点击。
2. 发送成功后收起面板、滚动到最新消息并清除发送状态。
3. 发送失败时提示错误、不插入虚假消息并恢复操作。
4. 选择面板使用静态压缩图；动态图片只在已发送消息中播放。
5. 下架或无法识别的历史表情显示“表情已下架”，不能显示破图。
6. 表情元数据统一由 `/api/stickers` 获取并缓存 30 分钟。
7. 图片使用懒加载和异步解码；翻页使用 `transform: translate3d(...)`。
8. 加载中使用固定 4 × 2 骨架；空状态显示“暂无可用表情”。

## 4. 昵称、徽章图标与徽章名称

### 4.1 基准

海龟汤详情页的昵称后徽章展示是全局基准。所有“昵称＋徽章图标＋徽章名称”场景必须直接使用 `EquippedBadgeIcon` 的默认名称模式：

```tsx
<EquippedBadgeIcon badge={user.equippedBadge} className="h-4 w-4" />
```

`className` 只用于适配当前场景的图标尺寸。默认 `showName` 必须保持为 `true`。

### 4.2 名称样式

徽章名称由共享组件统一控制：

- 普通：蓝色
- 稀有：紫色
- 史诗：金色
- 传说：传说渐变文字
- 字号：11px
- 字重：`font-black`
- 图标与名称间距：共享组件默认间距
- 名称不换行

禁止页面自行添加固定颜色的徽章名称，禁止在共享组件外再次渲染 `badge.name`。

### 4.3 布局规则

1. 昵称、徽章图标和徽章名称应在同一身份行内展示。
2. 昵称可以截断；徽章整体保持为不可拆分的 `shrink-0` 单元。
3. 不得把徽章名称另起一行，也不得在状态行中重复徽章名称。
4. 在线/离线、主持人、玩家等身份状态与徽章分开表达。
5. 密集列表和聊天消息默认关闭徽章动画，避免同时播放大量动画。
6. 只有明确的纯图标场景可以使用 `showName={false}`，例如：
   - 个人主页的大号徽章装饰
   - “我的”徽章装配按钮
   - 明确没有名称展示空间的纯图标操作
7. 使用 `showName={false}` 时，不能再在别处手写一个不同样式的徽章名称。

### 4.4 当前适用页面

- 海龟汤详情作者与评价者
- 首页海龟汤卡片和用户搜索结果
- 玩汤房间成员与聊天消息
- 圈子聊天消息、成员列表和 `@成员` 候选
- 后续任何昵称旁展示徽章名称的位置

## 5. 圈子 `@用户`

### 5.1 输入与选择

1. 长按其他成员头像，将完整的 `@用户昵称` 插入输入框。
2. 输入 `@` 加文字时，只匹配当前圈子成员的昵称。
3. 候选列表展示在输入框上方，交互参考微信。
4. 一次最多展示 5 个匹配成员。
5. 点击候选成员后，用完整昵称替换当前 `@查询文字`，并在昵称后保留空格。
6. 候选行展示头像、昵称、统一徽章图标与徽章名称、在线状态。
7. 表情包键盘打开时出现有效 `@` 查询，应关闭表情包键盘；打开表情包键盘时关闭 `@` 候选。
8. 可触发 `@用户` 的头像必须禁用 WebKit/微信的图片长按菜单、文字选择和图片拖拽，避免出现“转发给朋友、保存到手机、收藏、搜一搜”等原生操作；限制只作用于该头像触发器，不得全局禁止图片长按。

### 5.2 消息展示

1. 消息结构为：头像＋昵称＋徽章图标＋徽章名称＋消息。
2. 对被 `@` 的当前用户，消息中的完整 `@当前用户昵称` 显示为蓝色。
3. 不是当前用户的 `@昵称` 不变蓝。
4. 圈子消息一次最多加载 100 条；更早消息继续按游标分段加载。

### 5.3 未读 `@` 定位

1. 圈子存在未读 `@自己` 时，圈子列表的摘要优先显示：
   - `有人@你 [消息内容]`
2. 进入圈子后，右下角显示圆形 `@` 按钮及剩余数量。
3. 点击按钮后按需分段加载历史消息并滚动到对应消息。
4. 每处理一条就标记该条 `@` 已读；全部处理后按钮消失。
5. 离开圈子页面时，剩余未读 `@` 全部标记为已读。
6. 当前页面不在最底部时显示向下圆形按钮；点击后平滑滚动到最新消息。

## 6. 圈子未读提示

### 6.1 圈子列表

1. 已加入圈子的未读消息数量显示在圈子头像右上角。
2. 使用红色圆形数字标记，最多显示 `99`，超过后显示 `99+`。
3. 未读数量不写入卡片正文，避免与最新消息重叠。
4. 最新消息必须截断并使用省略号，不能与其他信息重叠。
5. 未读 `@自己` 摘要优先于普通最新消息摘要。

### 6.2 底部圈子导航

1. 当前不在圈子列表页 `/circles` 且存在普通圈子未读消息时，圈子图标右上角显示红色小点。
2. 存在未读 `@自己` 时，显示与红点占位相同大小的红色 `@`。
3. 未读 `@` 的优先级高于普通未读红点；两者不能同时显示。
4. 位于圈子列表页时不显示底部图标标记。
5. 新消息、进入圈子标记已读、读取单条 `@` 和离开圈子清空 `@` 后必须实时刷新。
6. 圈子未读数不计入顶部“消息”铃铛的总数。

### 6.3 顶部 `@` 横幅

1. 用户不在对应圈子聊天页面时，被 `@` 后显示全局顶部横幅。
2. 横幅的视觉、出现动画、停留时间、点击行为和滑动关闭行为与私信横幅一致。
3. 横幅展示发送者头像、昵称、“圈子@消息”和消息摘要。
4. 点击横幅进入对应圈子，并定位到该条 `@` 消息。
5. 当前已在对应圈子页面时不重复显示横幅。

## 7. 在线状态

1. 在线状态统一使用头像右下角绿色圆点。
2. 绿点必须有白色描边，避免与头像内容混在一起。
3. 适用于：
   - 圈子成员及 `@成员` 候选
   - 私信会话和聊天对象
   - 关注列表
   - 粉丝列表
   - 管理后台用户列表的最后登录时间字段
4. 在线状态优先使用实时事件更新；低频轮询只作为断线兜底。
5. 在线文字状态与徽章名称分开，不得把两者拼成同一个身份名称。
6. 管理后台“最后登录时间”字段对在线用户显示“在线”，离线用户显示最后登录时间；按最后登录时间倒序排序时，在线用户优先，其余用户再按最后登录时间倒序排列。

## 8. 玩汤房间消息滚动

1. 用户位于消息列表底部时，新消息继续自动跟随到底部。
2. 用户上滑查看历史消息后，新消息不得抢占当前位置或自动拉到底部。
3. 用户不在底部时，在“提示/线索”或主持人“更多”圆形按钮左侧显示同尺寸的圆形向下箭头按钮。
4. 点击向下箭头后平滑滚动到最新消息并隐藏按钮；手动滚回底部后按钮也应隐藏。
5. 用户主动发送文字或表情成功后，仍应回到最新消息。
6. 打开表情包键盘时，向下箭头与同组悬浮操作按钮保持一致的隐藏动画。
7. “线索/进度”弹窗必须由 Modal 外层承担唯一的纵向滚动，卡片列表不得再创建嵌套滚动区域；卡片区域使用稳定、隔离的渲染层，避免微信 WebView 滑动时闪烁。
8. “线索”和“进度”卡片均按时间倒序展示，最新内容位于顶部；线索编号和正式提问编号保持原始顺序编号，不随展示顺序重新编号。
9. 推理阶段内，主持人掉线不暂停正式提问；玩家可以继续发送讨论和正式提问，问题正常编号并进入进度列表，等待主持人重新连接后回复。旁观者权限和非推理阶段限制保持不变。
10. 主持人连续离线达到 30 分钟后，服务端自动解散房间：房间立即从大厅移除、禁止再次进入，并向仍连接的成员广播房间关闭事件。清理任务每分钟执行一次，因此关闭最多允许约 60 秒调度延迟。
11. 多条实时消息通知并发到达时必须合并并在当前增量请求结束后再次补拉，禁止直接丢弃后续通知；WebSocket 每次连接或重连成功后立即补拉断线期间的消息。
12. WebSocket 心跳连续 75 秒未收到服务端响应时主动断开重连；页面可见期间每 15 秒执行一次低频消息对账，页面从后台恢复时立即对账，保证代理丢帧或移动网络半断开时消息最终及时可见。

## 9. 导航与返回

1. 从圈子聊天点击用户头像进入用户详情，返回时必须回到原圈子详情页。
2. 从圈子或私信邀请卡片进入玩汤房间，房间不存在、已关闭或确认退出时，应返回邀请来源聊天页。
3. 需要保留来源的跨模块跳转必须通过明确的路由 state 或受校验的返回路径实现，不能默认返回首页。
4. 私信或圈子中的海龟汤分享必须使用统一的横向卡片；分享消息容器固定占聊天内容宽度的 78%，卡片最大宽度为 390px，不得按内部文字或图片的最小宽度收缩；封面区域约占卡片宽度的 38%，高度铺满卡片，图片居中等比裁切并完整覆盖封面区域，不得出现留白；卡片同时展示名称、热度、作者、标签、摘要、评分、难度、点赞和收藏。
5. 点击海龟汤分享卡片直接进入对应详情页；详情页返回时必须回到发起跳转的原私信或圈子聊天页。
6. 海龟汤详情页分享入口常驻；分享到微信时发送汤面图片，分享到私信或圈子时发送可点击的海龟汤卡片，不在普通文字消息中拼接链接代替。
7. 玩汤房间二维码必须使用当前站点同源邀请链接，并在未登录时保留房间 ID 与邀请令牌，登录成功后继续加入流程。HTTP 部署的认证 Cookie 必须配置 `COOKIE_SECURE=false`；登录界面必须通过 `/api/auth/me` 二次确认 Cookie 已保存后，才能更新为已登录状态。

## 10. 验收清单

- 所有表情包入口均为键盘式面板、4 × 2、可横滑且不覆盖消息。
- 玩汤、私信和圈子复用同一个 `StickerKeyboard`。
- 所有昵称后的徽章名称都由 `EquippedBadgeIcon` 默认模式渲染。
- 页面中不存在手写固定颜色的已装配徽章名称。
- 圈子 `@` 候选最多 5 人，只匹配当前圈子成员。
- 未读 `@` 可从圈子列表、顶部横幅和详情页按钮正确定位并消除。
- 普通圈子未读显示底部红点，未读 `@` 显示红色 `@`，且 `@` 优先。
- 在线头像绿点在规定页面中保持一致。
- 所有聊天历史加载、失败反馈、图片压缩和缓存策略符合本文档。
- 私信与圈子的海龟汤分享卡片结构一致，点击进入详情后可准确返回来源聊天页。

## 11. 活动徽章与限定徽章

1. 新建活动徽章或限定徽章时，只创建名称、描述、获取方式、图片、等级和成就点等基础资料，不同时配置或推导发放条件。
2. 新建活动徽章的 `activity_conditions` 默认必须为空；仅在后续收到明确的活动规则配置需求时，才允许通过后台单独设置自动计算与发放条件。
3. 活动徽章按当前条件向从未获得过该徽章且符合条件的用户自动发放；条件变更不得收回既有用户的徽章，也不得向历史上已经获得过该徽章的用户重复发放。
4. 限定徽章不配置自动发放逻辑，只允许管理员在管理后台直接发放和回收。

## 12. 贝壳货币与每日任务

1. “我的”页面在昵称下方固定展示当前贝壳余额和“明细”入口；任务中心放在金刚区，与现有功能入口保持相同尺寸、层级和点击反馈。
2. 任务中心展示当前余额、今日实际获得数量、每日 30 贝壳上限和全部任务进度；任务奖励自动到账，不设置手动领取按钮。
3. 每项任务必须同时展示单次奖励、当日进度、完成状态和当日实际获得数量；达到每日总上限后，任务仍可完成并记录进度，但实际获得显示为 0。
4. 贝壳明细按时间倒序展示变动类型、变动数量、变动后余额和时间；增加使用正数与绿色，扣减使用负数与红色。
5. 管理后台用户列表中的贝壳余额使用蓝色可点击样式；点击后展示该用户贝壳明细，并提供“增加”和“扣减”操作。
6. 贝壳数量只允许整数。后台扣减不得超过当前余额，所有调整完成后必须立即刷新余额和明细。
7. 从任务中心进入贝壳明细时，返回必须回到任务中心；从“我的”余额入口直接进入贝壳明细时，返回“我的”，不得丢失入口来源。
